# Application Architecture & Development Rules

## 1. System Framing
- **`umbra` is a Platform SDK & Application Harness (SOE):** It is an opinionated framework that encapsulates layout architecture, runtime execution bounds, design tokens, and spacing invariants under a single contract.
- **DO NOT** treat `umbra` as just a primitive button/input library. It owns the application structure.

---

## 2. Mandatory Layout & Vertical Rhythm Rules
Claude MUST follow these non-negotiable spacing rules to prevent random layout guessing:

* **NO OUTER MARGINS ON TEXT:** **NEVER** add `mt-*`, `mb-*`, or `my-*` classes to individual headings (`h1`-`h6`), paragraphs (`p`), or labels (`label`). Text height is purely line-height.
* **CONTAINERS OWN ALL SPACING:** All vertical and horizontal rhythm must be controlled strictly via flex/grid container `gap-*` utilities.
* **STANDARD SPACING SCALE:**
  - `gap-1` or `gap-2` (4px–8px): Tight pairings (e.g., Heading + Subtitle, Label + Input).
  - `gap-4` (16px): Content stacks (e.g., form fields, paragraphs inside a body section).
  - `gap-6` or `gap-8` (24px–32px): Major structural sections, card padding, dialog sections.

---

## 3. Component Architecture & Smart Defaults
* **Slot Composition:** When creating structural modules, expose slot props (`children`) for dynamic content while hardcoding interior layout gaps.
* **Smart Defaults:** Pre-configure common actions (e.g., default button labels), allowing consuming apps to override props only when necessary.

---

## 4. Workflow: Local First, Promote Later
* **Step 1:** Always check `umbra` for existing shells, dialogs, or layout modules before writing custom UI.
* **Step 2:** If a layout or component does not exist in the SDK, build it **locally inside the app**.
* **Step 3:** Write local custom components as clean, modular blocks so they can easily be promoted to `umbra` later.
